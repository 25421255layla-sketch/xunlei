CHAPTER 02

三家产品深度拆解

每家按统一的 6 段式：产品定位 / 产品架构 / 核心功能 / 数据链路 / 亮点与局限 / 可借鉴清单。不抄原文，只翻译产品思路。

2.1 阿里 Quick Audience

2.1.1 产品定位

Quick Audience（简称 QA）是阿里云 Dataphin 体系下的全渠道消费者运营平台。它和达摩盘有亲戚关系——达摩盘是阿里妈妈电商广告侧的 DMP（强绑淘系公域），QA 则是把同一套方法论搬到了阿里云，做成企业可私有部署的 CDP 版本，对接企业自有业务。

QA 的产品哲学可以用一句话概括："以受众为中心的一站式消费者资产管理和运营平台"。它的用户目标非常明确——市场营销人员和运营人员，而不是数据分析师或开发。

2.1.2 产品架构

QA 的产品架构从上到下分为 5 层，和达摩盘有非常强的同源感：

营销活动层面向业务人员
主活动/子活动 活动日历 营销任务编排 活动复盘
用户触达层多渠道激活
阿里妈妈（达摩盘） 品牌数据银行 瓴羊超信（短信） Kafka/API 下发 Salesforce 集成
用户洞察层分群与分析
受众圈选 标签透视分析 RFM 模型 AIPL 模型 显著性分析 行为筛选
数据建模层模型与集合
用户标签数据集 AIPL 模型配置 RFM 模型配置 行为数据集
数据接入层数据源与数据集
AnalyticDB MaxCompute Dataphin QuickTracking（埋点）

2.1.3 核心功能（4 大块）

数据接入 / 建模

让运营人员"无需懂数据"

QA 最大的克制是：不做 ETL。它把数据建模留给 Dataphin 做，自己只做"数据集"概念——运营人员把一张标签宽表注册为"数据集"即可使用。这种"职责分离"让产品非常轻。

支持的 3 种核心数据集：标签数据集、AIPL 模型数据集、RFM 模型数据集、行为数据集。

用户洞察

方法论驱动的分析框架

透视分析 + RFM 分析 + AIPL 用户分析 + AIPL 流转分析 + 显著性分析。注意：QA 把 AIPL 流转分析这种"业务专用分析"做成了内置功能，而不是让用户自己建 BI 报表——这是强方法论产品的典型做法。

受众管理

一切以"受众"为中心

QA 里"人群"叫受众（Audience）——品牌营销语境下更精准。受众支持 4 种圈选方式：标签筛选 / RFM 筛选 / AIPL 筛选 / 行为筛选。支持并/交/差运算，支持预估人数。

受众可直接关联到"营销活动"，形成"活动 → 受众 → 任务"的业务主线。

营销活动

整合业务语义的独特创造

QA 把"营销活动"做成一个独立主模块，而不是散落在触达任务里——这个做法在三家里独此一家。主活动（如"618 大促"）下挂子活动（如"品牌日"），子活动下挂受众 + 营销任务。

这种组织方式让"复盘"变得简单：点一个活动就能看所有受众 + 所有任务的效果。

2.1.5 亮点与局限

亮点
  • 方法论落地最彻底——AIPL 和 RFM 不仅是分析工具，而是贯穿整个产品的"世界观"
  • "受众"概念精准——比"人群"更有业务语境，适合营销语言
  • 营销活动作为一级模块——把分散的触达任务收束到"活动复盘"主线
  • 门店运营 + 地理圈人——QA 独创能力，适合有线下门店的品牌
  • 显著性分析——对比大盘找差异特征，非常精准的洞察工具
局限
  • 重品牌、重电商——整个产品假设用户做品牌营销，对迅雷云盘这种订阅 SaaS 不太贴
  • 不做 ETL——对数据中台成熟的公司是优点，但 MVP 小团队若没有 Dataphin 背景会觉得“数据进不来”
  • AI 化相对保守——2024 年才推复购预测模型，对比腾讯的 Agent 化进展慢
  • 实时能力弱——整体偏离线批处理为主，实时规则引擎是后来补上的
  • 强绑阿里生态——离开达摩盘、品牌数据银行、瓴羊超信后价值打对折

2.1.6 对迅雷云盘的借鉴清单

可以抄
① 受众圈选的 UI（左侧标签树 + 右侧条件构建器 + 底部预估人数）是行业标杆；
② "活动-受众-任务" 三级组织方式可以改造为"营销场景-人群-推送任务"；
③ 显著性分析（对比大盘找差异）这类功能可以做成画像洞察里的"一键洞察"；
④ RFM 模型的可视化配置交互（自定义得分规则和阈值）做得非常成熟。
不要抄
AIPL 模型不适合云盘业务——阿里的 AIPL 假设用户有"品牌认知 → 购买"心理路径，云盘用户关注的是"能否解决文件存储问题"。建议为云盘业务自建 RAPRC 模型（Register-Activate-Pay-Renew-Churn）。

2.2 腾讯 企点 CDP

2.2.1 产品定位

腾讯企点 CDP 诞生于 2021 年，是企点营销云的数据基座。根据 IDC 报告，2024H2-2025H1 期间，它已成为中国 CDP 市场份额第一。产品逻辑和 QA 差别最大的地方在于：它不是为营销人而生，而是为多角色、集团型企业设计。

它的核心卖点是两个词："实体化 + 智能体"。实体化意味着它能处理不只是"人"，还包括"货 / 场 / 导购 / 门店"等多种业务实体；智能体意味着 2024 年起它在 AI Agent 方向投入最激进。

2.2.3 核心功能特色

标签引擎

10+ 种生产方式最克制

官方列出的标签生产方式：规则标签 / 偏好标签 / 计算标签 / 首末次标签 / RFM 标签 / SQL 标签 / 外部标签（API 写入）等。这个分类体系比字节（十余种）更清晰、比阿里（主要是筛选式）更完整。

关键细节：每种标签生产方式都有独立的配置向导，而不是一个通用表单做十件事——这是 B 端产品易用性的决定性差异。

多实体能力

只有它原生支持"非人"实体

三家里只有腾讯 CDP 原生支持"人货场"多实体。不仅能给"客户"打标签，也能给"商品""门店""导购"打标签，实体间可建立关联关系，支持"关联查询"式的分群（如：找"消费了某品类商品且由某导购服务过"的客户）。

对云盘的启示：我们可以把"资源（视频/文档/软件）""设备""场景（家庭/办公）"建模为多实体。

AI Agent

2024 年起 4 大 Agent 组合

腾讯在 CDP 里做了 4 个明确职能的 Agent：人群圈选、权益设计、智能选品、文案生成。它们不是独立产品，而是嵌入在 CDP 工作流里——比如圈人时自然呼出“圈选 Agent”，用对话式交互完成。

这是AI-Native CDP 而非 “AI 贴在侧边栏” 的代表。

2.2.5 亮点与局限

亮点
  • 多实体能力独一档——原生支持人货场，扩展性最强
  • 标签引擎最优雅——10+ 生产方式分类清晰，向导式配置
  • AI Agent 组合拳——4 大 Agent 是当前行业最激进的 AI 落地

局限
  • 产品复杂度高——功能齐全但学习曲线陡，中小团队自建很难驾驭这个产品复杂度

2.2.6 对迅雷云盘的借鉴清单

重点抄
① 10+ 标签生产方式的分类体系——每种生产方式独立向导，这是我们标签生产中心原型的核心设计；
② 多实体扩展——云盘也有"资源 / 设备 / 订阅周期"等非人实体，架构层面预留扩展能力；
③ 客户 360 视图的"身份信息"独立模块——对合规敏感的云盘尤其重要，要让运营能看到为什么这些 ID 属于同一个人。
谨慎抄
AI Agent 组合拳很惊艳，但我们 MVP 阶段不应追——先让基础能力跑起来，Agent 放在 V4/V5 规划；"企微生态 + 私域"玩法与云盘业务无关，不用抄。

2.3 字节 火山 VeCDP

2.3.1 产品定位

VeCDP 是火山引擎数智平台（VeDI）旗下的 CDP 产品，脱胎于字节跳动内部的数据中台。它和阿里腾讯最大的不同是：产品重心在数据工程能力，而不是业务方法论。白皮书里它花了 60% 篇幅讲架构、性能、ETL 算子，只有 20% 讲业务场景。

这决定了它的用户画像是："有一定数据思维的运营人员 + 数据工程师"的混合团队。它假设你公司有数据人员，但不希望你完全依赖数据团队。

2.3.2 产品架构

VeCDP 公开的产品架构是三家里最清晰的——数据层 → 汇聚层 → 整合层 → 应用层四段式，加上项目中心 + 运维监控两个基础设施：

2.3.3 核心功能特色

可视化建模

把 ETL 做进 CDP 里

这是 VeCDP 独此一家的设计：CDP 产品里内嵌了一个简化版 ETL 工具。用户拖拽算子（字段设置 / 筛选行 / 添加计算列 / 聚合 / 连接 / 合并 / 行列转置…）就能把原始数据加工成可用的数据集，不需要去 Dataworks 或 Dataphin 做。

优点：数据加工权限可以下放给业务侧的数据分析师，缩短需求链路；缺点：产品复杂度陡增，对非技术用户是劝退点。

实时规则引擎

事件驱动的营销自动化

监听标签、行为、分群的实时变化，满足条件时以 Kafka 消息形式推送给下游系统。典型场景：用户下单失败 10 秒内触发挽留弹窗、用户会员到期前 7 天触发续费提醒。

这种“实时信号 → 下游业务”的机制和传统“分群推送”是两套完全不同的触达范式，弹性高得多。

ID 图谱

最透明的 OneID 设计

VeCDP 把 OneID 叫做 Baseid，以图谱形式可视化——一个 OneID 节点周围连接手机号、设备号、微信 UnionID 等各种 ID。支持离线/实时双数据源 ID 融合，可配置 ID 优先级、是否允许 OneID 变更。

关键体验点：产品把"为什么这些 ID 属于同一个人"可视化了——这在合规审计和业务信任上极其重要。

算法模型

开箱即用的 4 大行业模型

Lookalike（私域相似人群）、聚类模型、回购预测、流失预警。这些模型都做成了表单式向导——业务人员填几个参数就能跑，不用调参也不用写代码。

背后用字节自己的算法平台实现，属于"算法能力产品化"的典范。

2.3.5 亮点与局限

亮点
  • 架构最清晰——四层产品架构 + 基础设施是 CDP 教科书式的架构样板
  • 可视化建模独此一家——把 ETL 做进 CDP 里的唯一产品
  • 实时能力强——实时规则引擎 + 实时标签 + 实时 ID Mapping
  • 性能明确——ByteHouse 支持千亿级秒级查询，数据公开
  • 模型产品化——4 大行业模型开箱即用，这是小团队自建最难复制的能力
局限
  • 工程气息重——业务人员进入门槛高，需要对"数据集""算子""DSL"有感知
  • 缺少业务方法论——不像阿里有 AIPL、RFM 作为业务主线
  • AI 进度滞后——对比腾讯 Agent 组合慢了一步，AI 主要在算法模型层
  • 营销应用层薄——触达主要靠打通抖音巨量引擎，不适合不玩广告的企业

2.3.6 对迅雷云盘的借鉴清单

重点抄
① 四段式架构——数据层/汇聚层/整合层/应用层，作为我们产品架构的顶层设计蓝本；
② ID 图谱可视化——用 SVG 把 OneID 合并关系画出来，对客服和合规审计是硬需求；
③ 实时规则引擎——云盘业务事件丰富（会员到期、扩容、活跃）“事件驱动触达”比分群推送更实用；
④ 4 大行业模型的“向导式配置”——流失预警、回购预测对云盘业务完美匹配（改名：“续费预测”“流失预警”“扩容意向”）。
不要抄
可视化建模（ETL）虽然惊艳，但 MVP 阶段坚决不做——你们已有成熟数据仓库，数据建模放在数仓层更合理，CDP 只做“注册数据集”这一层即可。自建 ETL 会让 MVP 研发周期翻倍。

· 用阿里的"受众+活动"作为业务语义层（云盘的"运营活动"天然贴合此模式）。

4.2 每个环节的深度拆解

01
数据接入 — Ingestion

做什么：把散落在各处的原始数据（埋点日志、业务数据库、第三方 SaaS）汇聚到统一数据层。

输入：MySQL/PostgreSQL（会员库、订单库）、Hive/ClickHouse（数仓）、Kafka（实时事件）、CSV/Excel（运营导入）、API（第三方回调）

输出：Raw 层贴源数据 —— 保留原始结构，不做清洗，带上 ETL 时间戳

技术难点：(1) 增量同步（CDC/binlog）；(2) 实时与离线的统一写入；(3) Schema 变更兼容

产品决策点：自建 SDK 还是用开源（PostHog/Snowplow）？实时到什么程度（秒级 / 分钟级 / T+1）？数据源配置在产品内还是交给数据团队？—— 云盘建议：埋点用开源，实时分钟级，数据源配置放在产品内（业务自助）

02
数据建模 — Modeling

做什么：把 Raw 层数据清洗、转换、关联、聚合为可被业务使用的“数据集”。这一层是 CDP 向外露出的第一层"业务语义"。

输入：Raw 层多张原始表

输出：可被标签/分群/档案直接消费的结构化数据集（大宽表或明细表）

技术难点：(1) 数据质量校验；(2) 大宽表的存储与更新；(3) 元数据管理（字段血缘、业务含义）

产品决策点：最关键决策——ETL 做在 CDP 里（字节路线）还是做在数仓里（阿里路线）？—— 云盘建议走阿里路线：你们已有数仓，ETL 交给数仓，CDP 只注册“数据集”

03
ID 图谱 — OneID

做什么：把散落在不同渠道的用户标识（迅雷账号、手机号、设备 ID、微信 UnionID...）识别为同一个自然人，生成统一的 OneID（又叫 Baseid、PID）。

输入：各数据集中的 ID 字段（可能有几十种 ID 类型）

输出：ID → OneID 的映射表 + 每个 OneID 的属性快照 + ID 关系图谱

技术难点：(1) 图计算（Spark GraphX）；(2) OneID 稳定性（新数据来了要不要重新生成）；(3) 实时 vs 离线冲突

产品决策点：(1) 用图计算还是规则合并？(2) 单主体还是多主体？(3) ID 关系要不要可视化暴露给运营？—— 云盘建议：MVP 用优先级规则合并，V2 升图计算，ID 关系必须可视化（合规和客服排查必须）

04
数据档案 — Profile

做什么：以 OneID 为主键，聚合这个人的所有属性、行为、标签、所属人群、触达历史，形成“一个人的完整档案”。

输入：OneID 映射 + 所有数据集 + 标签结果 + 人群归属 + 行为事件流

输出：用户 360 视图（产品页面） + 档案 API（给业务系统调用）

技术难点：(1) 大宽表更新性能；(2) 行为时间轴的高效查询；(3) 敏感字段脱敏展示

产品决策点：档案页哪些信息放哪些不放？行为时间轴保留多久？云盘必须：文件内容绝对不进档案，行为元数据做好脱敏展示

05
标签生产 — Tagging

做什么：根据业务规则把“原始数据”加工为“业务可读的标签”。例如“最近 30 天下载次数 > 10”→“高活跃用户”。

输入：数据集 + 标签配置规则 + OneID 映射

输出：OneID → 标签值的键值对（存宽表或 BitMap）

技术难点：(1) 标签宽表的 schema 演进；(2) 增量计算 vs 全量重算；(3) 实时标签与离线标签的一致性

产品决策点：提供几种标签生产方式？怎么设计配置表单让业务人员自助使用？—— 建议参考腾讯的 10+ 种分类，MVP 先做 4 种（规则/首末次/RFM/导入）

06
分群圈选 — Segmentation

做什么：基于标签组合、行为条件、上传 ID 包等方式，把“所有用户”收窄为“一批目标用户”。

输入：标签宽表 + 行为事件 + 圈选规则（DSL 表达式）

输出：人群包（OneID 列表） + 元数据（规则定义、更新策略）

技术难点：(1) 亿级用户的秒级预估；(2) 人群的交并差运算（BitMap）；(3) 动态人群的定时更新

产品决策点：规则配置的 UI 是最关键的产品体验——必须支持并/交/差、可拖拽、实时预估，参考阿里 QA 的受众圈选 UI

07
画像洞察 — Insight

做什么：分析人群的特征分布、与大盘的差异、生命周期阶段、归因路径，为运营决策提供依据。

输入：人群包 + 标签宽表 + 业务模型（RFM/生命周期）

输出：可视化报告（图表/雷达图/桑基图/表格）

技术难点：(1) TGI 等统计指标的准确计算；(2) 大数据量下的可视化性能；(3) 洞察结果的业务解读

产品决策点：洞察是“分析师主动查”还是“系统自动生成结论”？—— 建议：MVP 做基础洞察，V4 阶段做 AI 一键解读（接入 LLM）

08
营销激活 — Activation

做什么：把人群包 / 标签值 / 实时信号输出到触达系统，真正触达到用户。

输入：人群包 + 触达模板 + 渠道配置

输出：发送到短信/Push/企微/广告系统的消息 + 效果回传

技术难点：(1) 触达通道的稳定性；(2) 频控/黑名单/合规；(3) 效果数据的归因回传

产品决策点：CDP 要不要自建触达能力？—— 建议：CDP 只负责“人群和信号输出”，触达接公司已有的 MA/Push 平台（不要重造轮子）

4.3 横向支撑能力（贯穿全链路）

治理

数据血缘 + 质量监控

从数据源 → 数据集 → 标签 → 人群的完整血缘追溯，出问题时能快速定位影响面。数据质量校验覆盖完整性、准确性、时效性、一致性、丰富性五大指标。

权限

四级权限模型

用户/用户组/项目/集团四级；字段级 + 行级 + 标签级授权；敏感标签脱敏展示。云盘业务建议采用“最小必要”原则——默认不可见，按需开放。

合规

《个保法》全链路合规

用户授权机制 + 敏感字段加密（MD5/AES）+ 人群导出审批 + 审计日志。云盘红线：文件内容不得作为画像数据源，只能用行为元数据。

Chapter Six

6. MVP 技术链路

本章面向技术负责人。目标是把前述产品设计翻译成一套"能开工"的技术方案——组件选型、数据流向、存储结构、关键 DSL 示例。前提假设：公司已有成熟数仓，CDP 不做重 ETL，走"数据集接入 → OneID → 标签/分群 → 应用"的轻链路。

6.1 组件选型（与现有数仓共存的方案）

对齐阿里 Quick Audience 的"数据集中心"思路：上游数据不搬家，CDP 只登记 schema，查询下推到 OLAP 引擎。这样 CDP 不用扛 ETL 重责，可以更快上线。

层级 推荐组件 职责 理由与替代方案
数据接入层 Kafka + Flink CDC / DataX 数仓 Hive 表 T+1 导入；MySQL 业务库 CDC 实时同步；客户端埋点 Kafka 直送 与现有数仓打通的最低成本路径。替代：DataPhin、DataWorks（如果已在阿里云上）
元数据 & 配置层 MySQL（主）+ Redis（缓存） 存标签定义、规则 DSL、人群定义、用户/权限/授权、推送任务 业务对象关系型为主，轻量级。标签定义不超 10 万行
OLAP / 查询引擎 ClickHouse（自建）或 ByteHouse（火山商业版） 承载标签宽表、行为明细、人群包预估 / 圈选 SQL 执行 核心是亿级用户秒级聚合。云盘用户量大，ClickHouse/ByteHouse 比 Spark 快 5-10 倍。替代：StarRocks、Doris
ID Mapping 计算 Spark + GraphFrames（离线）+ Flink（实时增量） 构建 ID 图，生成 OneID；每日全量 + 当日增量 连通分量算法（Connected Components）。数据量小于千万时可用 HBase + 递归。替代：图数据库如 Nebula（重）
标签 & 分群引擎 自研 DSL 编译器 → SQL（ClickHouse） 将可视化规则编译为 SQL；支持离线调度 + 实时规则触发 DSL 是核心护城河，直接用 SQL 暴露给业务不可行（学习成本、不可复用）。参考：字节 VeCDP 的 DSL 模式
算法平台 Python + 公司现有算法平台（训练）+ CDP 集成模型预测（推理） 续费预测、流失预警、Lookalike 由算法团队训练，CDP 负责导入结果作为标签 MVP 阶段算法不内建，只消费结果。V2 再考虑集成聚类等平台化算法
实时规则引擎 Flink + Redis（状态）+ Kafka（输出） 监听标签 / 行为 / 分群变化，触发信号推给下游 MA V2 引入。V1 只做离线每日调度即可
触达集成 人群包 → S3/OSS 落文件 + Kafka 信号 + API 推送 给下游客户端 Push 系统、微信服务号系统、站内信系统提供标准出口 人群包是输出品，CDP 不直接触达。解耦后各触达系统按各自 SLA 执行
存储 & 缓存 Hive (明细) + ClickHouse (宽表) + Redis (画像缓存) + OSS (人群包) 冷热分层：行为明细落 Hive，标签宽表落 CH，360 查询走 Redis，人群包落 OSS 分级存储控成本，Redis 扛 360 页面并发

6.2 核心数据模型设计

关键表设计遵循"高宽表 + 明细事实表"模式。标签宽表是 CDP 的心脏：一行一个 OneID，列是所有标签。

ID 图谱表 id_graph
字段：id_value（原 ID 值）/ id_type（类型枚举）/ one_id（生成的 OneID）/ first_seen / last_seen / source_ds
作用：所有 ID 到 OneID 的映射
量级：OneID 约 2 亿，原 ID 总数约 6-8 亿行
标签宽表 user_tags_wide
字段：one_id / member_level / expire_days / last_active_date / rfm_level / ... （~200 列）
分区：按 dt 日分区
量级：2 亿行 × 200 列，每日快照约 40GB（ClickHouse 压缩后）
行为明细表 user_events
字段：one_id / event_type / event_time / properties(JSON) / source
分区：event_date + event_type
量级：日增 20-50 亿行，只保留 90 天热数据在 CH，其余在 Hive
人群包表 segments
主表：segment_id / name / rule_dsl / owner / status / refresh_type
结果表：segment_users (segment_id, one_id, updated_at)
量级：典型人群 10 万-500 万，平台总计 2000 个活跃人群

6.3 关键链路：一个标签从定义到生效

以云盘「会员到期剩余天数」这个规则标签为例，走完全链路：

1. 业务在 UI 定义规则
2. 生成 DSL
3. DSL → SQL 编译
4. 调度器下发
5. CH 执行写回宽表
6. 标签生效

Step 1-2：UI 规则 → DSL

{
  "tag_id": "tag_expire_days",
  "tag_name": "会员到期剩余天数",
  "output_type": "int",
  "dataset": "ds_subscription_lifecycle",
  "update": { "type": "offline", "schedule": "daily 02:00" },
  "formula": {
    "op": "date_diff",
    "args": ["CURRENT_DATE", "subscription_end_time"],
    "unit": "day"
  },
  "filter": { "op": "AND", "children": [
    { "field": "subscription_status", "op": "=", "value": "active" }
  ]}
}

Step 3-5：DSL → SQL → ClickHouse 执行

-- 编译器输出（每日调度执行）
INSERT INTO user_tags_wide (one_id, expire_days, dt)
SELECT
    one_id,
    dateDiff('day', today(), subscription_end_time) AS expire_days,
    today() AS dt
FROM ds_subscription_lifecycle
WHERE subscription_status = 'active'
  AND dt = today() - 1;

-- 执行后元数据更新
UPDATE tag_meta
SET last_run_time = NOW(),
    user_count   = 12840000,
    status       = 'success'
WHERE tag_id = 'tag_expire_days';

6.4 关键链路：一次分群圈选的执行过程

人群圈选是 CDP 的最高频动作。对运营来说，他们期待"几秒返回命中数"，对技术来说，这要求 OLAP 层必须秒级响应。以下是"会员到期 7 天且近 30 日活跃"这个人群的执行过程：

-- 预估阶段：SELECT COUNT(DISTINCT one_id) 秒级返回
SELECT count(DISTINCT t.one_id) AS estimated_size
FROM user_tags_wide t
LEFT ANTI JOIN (
    SELECT one_id FROM user_tags_wide
    WHERE marketing_opt_out = 1 AND dt = today()
) ex USING(one_id)
WHERE t.dt = today()
  AND t.expire_days <= 7
  AND t.subscription_status = 'active'
  AND t.one_id IN (
      SELECT one_id FROM user_events
      WHERE event_date >= today() - 30
        AND event_type IN ('file_download', 'app_open')
      GROUP BY one_id
      HAVING countIf(event_type = 'file_download') >= 3
          OR countIf(event_type = 'app_open') >= 5
  );

-- 生成人群包阶段：物化到 segment_users 表
INSERT INTO segment_users (segment_id, one_id, dt)
SELECT 'seg_xxx', one_id, today()
FROM ( ... 同上 SELECT ... );

-- 导出给下游：推到 Kafka 或 OSS 文件
SELECT id_value
FROM id_graph
WHERE one_id IN (SELECT one_id FROM segment_users WHERE segment_id = 'seg_xxx')
  AND id_type = 'xunlei_uid';

6.5 合规设计：云盘场景的特殊约束

这是云盘 CDP 和电商 CDP 最大的差异点，也是系统设计的核心约束。
合规项 技术实现 影响
文件内容隔离 CDP 的输入数据源 白名单：只允许元数据 schema（扩展名、文件大小区间、MIME、上传下载时间、流量），文件名文本、内容 hash、哈希片段一律不接入 数据入口层就要有"schema 审批"机制，新数据源上线前合规评审
敏感字段脱敏 手机号、邮箱、身份证、支付账号存储使用 AES/MD5 双存（存 hash 用于 Join，存密文用于展示时解密）；UI 展示层强制脱敏中间位 所有画像查询走脱敏通道，原始值只在"导出为推送包"时短暂还原
授权状态校验 维护 marketing_opt_out 标签（从用户设置中心同步），人群圈选默认排除；此外每个渠道有独立的授权位（push_opt_out, wx_opt_out, email_opt_out） 分群引擎在"生成人群包"阶段强制 JOIN 授权表，拒绝未授权用户进入推送
数据访问分权 标签授权 + 数据规则权限（阿里/火山都做了）：每个标签对用户/用户组独立授权；财务相关标签只有 CFO 组可见 不能让运营看到全量支付金额，但可以看到"充值等级"这样的派生值
审计日志 所有 360 查询、人群导出、推送任务全链路记日志；敏感操作（导出明细）需二级审批 日志量大，建议独立的 ELK 集群承接
数据生命周期 行为明细 CH 保留 90 天，Hive 保留 2 年，超过归档或删除；账户注销后 T+30 内从画像中删除 响应《个保法》用户数据删除权

6.6 性能关键点：为什么云盘需要 ClickHouse/ByteHouse

上一份报告提到 VeCDP "亿级数据秒级响应" 的底气来自 ByteHouse。对云盘来说这不是锦上添花，而是刚需——原因如下：

场景 数据量 期望响应 Spark 实测 CH/BH 实测
人群圈选预估 2 亿用户 × 200 标签 < 3 秒 30-120 秒 1-3 秒 ✓
用户 360 查询 单 OneID 全标签 + 行为 < 500ms 不适合 100-300ms ✓
画像分布统计 某人群的标签透视 < 5 秒 60-300 秒 2-8 秒 ✓
离线标签计算 全量宽表刷新 可慢（2h 内） 1-3 小时 ✓ 可用但资源浪费

结论：CDP 的"在线查询层"必须是 OLAP（CH/BH/Doris/StarRocks 四选一），"离线计算层"可以继续用 Spark，两者是互补关系，不是替代。

7.2 改造后抄 · 需要本地化

这一档功能方向对，但三家的做法不能直接套，要根据云盘业务改。

功能 抄自 需要改造的点 改造后的云盘版本
生命周期模型 阿里 AIPL / 5A AIPL 是电商漏斗（认知→兴趣→购买→忠诚），云盘是订阅漏斗 自创 RAPRC：Register → Activate → Pay → Renew → Churn，每个阶段有专属标签和转化率指标
RFM 模型 阿里 / 腾讯 传统 RFM 的 M（货币）对云盘是"累计充值"，F（频次）对云盘更像"使用频次"而非"购买频次" R = 最近活跃日期；F = 近 30 日使用次数；M = 累计充值金额。保留 8 个传统象限但重命名
Lookalike 火山（种子人群扩展） 云盘没有广告投放场景，不需要扩展到公域 只做私域 Lookalike：基于"高续费用户"种子扩展到"未续费但特征相似"用户，用于续费挽回
实时规则引擎 火山（Kafka 信号输出） V1 不上，V2 再做。实时引擎对 Flink 运维能力要求高 V2 阶段上线。V1 用离线 T+1 + 分钟级刷新重点标签的"半实时"方案顶住
多实体 / 多主体 腾讯（人货场） 云盘没有"商品"，但有"设备"和"资源类别"（视频/软件/文档） 主体定义为「用户 + 设备 + 资源类别」三主体，主体间关系：用户-设备（使用）、用户-资源（偏好）
营销活动管理 阿里（活动日历 + 主子活动） 阿里按"618/双十一"大促节奏，云盘更像持续性续费/流失场景 改为"战役（Campaign）管理"：每个战役挂多个人群 × 多个渠道 × 多次推送，做归因统计
首页看板 火山 火山看板偏数据资产视角（标签数、人群数） 改成业务视角：会员规模、新增、流失、续费率、各生命周期阶段人数；分析师再加资产视角看板

7.4 不抄 · 不适合云盘

这一档是 场景根本不匹配，想抄也用不上。

功能 抄自 云盘为什么不做
达摩盘打通 / 品牌数据银行 阿里 QA 这是阿里电商生态内的公域回流通道，云盘没有等价场景
企业微信私域 腾讯企点 云盘是 2C 工具型产品，没有"销售加客户微信"这种私域玩法
巨量引擎 / 千川 回流 火山（字节系打通） 云盘不做广告投放场景（即使投放也不会大规模回流数据到 CDP）

货找人 / 商品标签 腾讯（多实体） 云盘没有"商品"这个主体，"空间包 / 会员套餐"的 SKU 极少（十几个），不需要做商品标签化

7.5 一句话借鉴总结

字节的架构 × 腾讯的标签引擎 × 阿里的受众组织方式，是 Pandora CDP 的最佳路径。

具体来说：
  • 骨架（分层架构）—— 抄字节：数据层 / 汇聚层 / 整合层 / 应用层，层次最清晰
  • 肌肉（标签与分群引擎）—— 抄腾讯：10+ 标签生产方式、多实体、DSL 底层
  • 皮肤（运营用的 UI）—— 抄阿里：受众圈选、AIPL/RFM 看板、营销活动管理的交互最成熟
  • 灵魂（业务语言）—— 自建：RAPRC 生命周期 + 云盘专属标签体系 + 合规红线设计
Chapter Eight

8. 迭代路线与风险

按"能用 → 好用 → 智能"三阶段规划，每阶段给出必做、可做、不做三个清单，以及该阶段最容易踩的坑。时间节奏参考业内成熟 CDP 建设周期。

8.1 V1 · MVP 阶段（预计 3-4 个月）

阶段目标：把数据管道跑通、让运营能自助圈一次人、让一次推送闭环。一切以"跑通"为标准，细节可以粗糙。

✓ 必做
  • 3-5 个核心数据集接入
  • OneID（迅雷 UID + 手机 + UnionID + UDID 四类）
  • 标签生产 4 类：规则 / 统计 / 生命周期 / 导入
  • 人群圈选（AND/OR/排除）+ 预估
  • 用户 360 基础版（概览 + 行为轨迹）
  • 3 个渠道推送：客户端 Push / 微信服务号 / App 弹窗
  • 数据脱敏 + 标签授权 + 频控
  • 10-20 个核心标签 Demo
◇ 可做（有余力再做）
  • A/B 测试能力
  • RFM 模型 / RAPRC 阶段标签
  • 偏好标签、首末次标签
  • 预测标签导入（算法团队提供）
  • 自定义看板
  • 基础画像分析（TGI 占比）
✗ 坚决不做
  • 可视化 ETL / AI 建模
  • AI Agent / 对话式分析
  • 实时规则引擎
  • 多主体复杂关系图谱
  • SQL 标签（给高阶用户的口子，V1 别开）
  • 跨项目 / 跨集团发布管理
V1 最容易踩的 3 个坑：
  1. OneID 做得太晚 —— 很多团队先做标签再回头做 OneID，导致标签全部要重算。正确做法：Day 1 就拉齐 OneID 方案
  2. 标签宽表列数失控 —— 上线 3 个月标签从 20 个涨到 300 个，宽表写入超时。预留标签分组 / 分表能力
  3. 触达渠道以为"对接完就 OK" —— 实际上每个渠道都有频控、授权、退订、落地页追踪的坑，要为每个渠道预留 2 周的联调

8.2 V2 · 能力深化（预计 4-6 个月）

阶段目标：从"能用"到"好用"。把 V1 打下来的地基上面盖两层楼——实时能力、高阶标签、洞察分析。

✓ 必做
  • 实时规则引擎（Flink 接入）
  • 组合标签 / 逻辑标签 / SQL 标签
  • 多主体：用户 + 设备 + 资源类别
  • 画像洞察：TGI 分析、显著性分析、下钻
  • 人群交并差再加工
  • 营销效果分析闭环（归因）
  • 标签血缘 + 数据质量监控
  • 战役（Campaign）管理
◇ 可做
  • 私域 Lookalike（种子扩展）
  • 聚类分群（算法驱动）
  • 多维特征分析（自动挖掘）
  • Open API（外部系统取数）
  • 自定义指标中心
✗ 先不做
  • AI Agent 多 Agent 协作
  • 大模型对话式分析
  • 自动化营销编排（Journey Builder）
V2 最容易踩的 3 个坑：
  1. 实时引擎引入过早 —— Flink 运维复杂度陡增，没有足够业务诉求支撑就上，会被运维压垮。必须有至少 3 个"必须实时"的业务场景才引入
  2. 归因做不出来 —— 多渠道触达后业务方一定会问"这波续费是哪个渠道拉回来的"，但归因规则（首因 / 末因 / 线性）业务经常说不清，要提前对齐
  3. 标签爆炸后的治理 —— 300+ 标签时，重复标签、僵尸标签、口径漂移都出现。要引入"标签健康度"概念：覆盖率 < 1% 或最近 90 天无引用的标签自动下架

8.3 V3 · AI Agent 化（预计 6 个月+）

阶段目标：从"好用"到"智能"。让 AI 成为运营的副驾驶——帮他想标签、帮他圈人、帮他写文案。对标腾讯营销云目前的智能体方向。

✓ 必做
  • 人群圈选 Agent：自然语言 → DSL
  • 文案生成 Agent：根据人群特征出文案
  • 自动化营销 Journey（触发器 + 编排）
  • AI 辅助标签推荐（你可能漏了这些标签）
◇ 可做
  • 对话式数据分析
  • 智能归因分析
  • 选品 / 权益推荐 Agent
  • 异常检测 Agent（数据流异常、口径漂移）
✗ 不要先跑得太快
  • 不要把人完全替换 —— Agent 是辅助
  • 不要跳过 V1/V2 直接上 V3
  • 不要盲目跟风所有 AI 能力
